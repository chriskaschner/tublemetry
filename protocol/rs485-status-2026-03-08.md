# RS-485 Hot Tub Project — Status as of 2026-03-08

## Project Goal

Automate hot tub temperature setpoint via Home Assistant to execute TOU rate
optimization (drop to 99°F on-peak, reheat to 104°F before evening soak).
Eliminates twice-daily manual adjustments, saves ~$10/month on MGE Rg-2A rate.

---

## Hardware

- **Hot Tub:** Strong Spas Rockport S6-0001 (Costco "Evolution" / "Convertible Spa")
- **Control Board:** Balboa VS300FL4 (PCB: VS500Z, P/N 22972_E2)
- **Topside Panel:** VL-series (VL401/VL403/VL406U) — "dumb" panel, no microcontroller
- **Electrical:** 240V, 4kW heater, 2BHP two-speed pump
- **Connection:** RJ45 between board (J1) and topside panel
- **Tap Method:** Monoprice RJ45 T-splitter at J1, sacrificial Cat5 to Wago breakout

- **Multimeter:** Orange/White to Blue = 5V (power/ground identification)
- **Logic analyzer** (Lonely Binary 8ch, PulseView/Sigrok): All 8 pins captured

  all others as steady-state. Captures: `254.sr`, `OH.csv`
- **RS-485 adapter** (Waveshare USB, SP485EEN): Decoded UART at 115200 8N1 on
  Pin 5 (board→panel heartbeat) and Pin 6 (display refresh stream)
- **Multimeter button testing:** Each button pin measured while pressing all 4 buttons

## Button Input — Analog Voltage Matrix

Buttons are NOT digital/UART. The board reads 4 analog pins via ADC.
Each button press pulls exactly ONE pin from ~2.3V idle to ~4.7V.

| Button    | Spike Pin               | Idle   | Pressed |
|-----------|-------------------------|--------|---------|
| Temp Up   | Pin 2 (Orange)          | 2.82V  | 4.69V   |
| Temp Down | Pin 8 (Brown)           | 2.26V  | 4.70V   |
| Lights    | Pin 3 (Green/White)     | 2.27V  | 4.71V   |
| Jets      | Pin 7 (Brown/White)     | 2.26V  | 4.71V   |

### Manual button simulation: CONFIRMED WORKING
- Briefly bridging Pin 1 (+5V) to Pin 8 (Temp Down) successfully lowered the
  displayed temperature setpoint by 1°F
- This proves an ESP32 can inject button presses by driving the correct pin high

### Sensitivity warning
- The analog button lines are sensitive to loading/noise
- Even an unterminated Cat5 breakout cable caused phantom Temp Up presses
- ESP32 circuit MUST use high-impedance isolation when idle (analog switches
  or MOSFETs, not bare GPIO)

## RS-485 Data Streams (Board → Panel)

### Bus Parameters
- **Baud rate:** 115,200
- **Data format:** 8N1
- **Protocol:** Balboa GL/ML display protocol — NOT standard BWA `0x7E`-framed
- **Architecture:** Two simplex RS-485 channels (both board→panel), NOT half-duplex
- **Refresh rate:** 60Hz (16.7ms period) — both streams synchronized

### Protocol type
The VS300FL4 with VL-series topsides uses the **Balboa GL/ML display protocol**.
VL panels are "dumb" — no microcontroller. The board directly drives the panel's
7-segment displays and LEDs via continuous serial data. Reference project:
`github.com/netmindz/balboa_GL_ML_spa_control`

### Display Multiplexing (from OH.csv timing analysis)

The two RS-485 streams are **synchronized display multiplexing signals**, not
independent data channels:

1. **Pin 6 (refresh)** fires first — selects which digit position to illuminate
2. **Pin 5 (content)** fires ~400µs later — sends segment data for that digit
3. Both repeat at 60Hz (16.7ms period)

| Stream | Burst duration | Estimated bytes/burst | Behavior during OH flash |
|--------|---------------|----------------------|--------------------------|
| Pin 6 (refresh) | 1055µs | ~12 bytes | **Continuous** — never stops (599 bursts/10s) |
| Pin 5 (content) | 572µs | ~6-7 bytes | **Only during "on" phase** — silent during "off" (298 bursts/10s = 50% duty) |

**OH flash timing:** ~500ms on + ~500ms off = 1Hz. During "off" phase, Pin 6
keeps refreshing (maintaining digit selection) but Pin 5 goes silent (no segment
data = display blank). This is how the board makes the display flash.

### Pin 5 (Blue/White) — Display Content Stream

**Idle/Heartbeat Frame:**
```
FE 06 70 XX 00 06 70 00
```
8-byte repeating frame, continuous, ~110ms between chunks. These bytes are
7-segment bitmap data encoding the current temperature display. The second
half (`00 06 70 00`) appears to be a duplicate sub-frame for a second panel port.

**Byte index 3 varies with temperature:**
- `0x30` — observed in hex.txt (TeraTerm capture, ~1350 frames)
- `0xE6` — observed in rs485_jets.txt (Python capture, 285 frames, stable)
- `0x38` — 5 sporadic occurrences in hex.txt, likely noise/transition artifacts

**Button Press Responses (board status changes on this stream):**

- **Temp Down** (~2s delay): 11-byte burst, then modified frame `06 70 E6 00 00 06 00 F3`
- **Temp Up** (~0.5s delay): same burst pattern, then modified frame `06 70 E6 00 00 03 18 00`
- **Lights** (~0.5s delay): `FE` byte drops from frame (7 bytes instead of 8)
- **Jets:** No change detected — confirmed across 8 button presses in 30s capture (rs485_jets.txt)

**Button-press burst pattern (11 bytes):**
```
E6 77 [DIGIT] E6 77 E6 E6 77 E6 E0 FF
```
Captured in hex.txt at frame ~1346. The `[DIGIT]` byte (position 2) matches
the current temperature digit from the idle frame — e.g., `0x30` when idle
frame shows `0x30` at byte index 3. This confirms the burst adapts to the
current temperature. The burst likely triggers the panel's "setpoint flash"
display mode.

**Display mode transitions (from teraterm.txt scan, 1610 lines):**

Three distinct display modes found, plus the idle temperature display:

| Mode | Byte pattern (replaces `70 30` in idle) | Lines | Duration |
|------|----------------------------------------|-------|----------|
| **Idle (temperature)** | `70 30` or `70 E6` | 1-680, 725-735, 1290-1610 | Majority of capture |
| **Button press (setpoint flash)** | `77 [various]` — `0x77` replaces `0x70` | 681-724, 736-1153 | ~470 lines, two blocks |
| **Pr mode (priming?)** | `67 33` replaces `70 30` | 1213-1289 | ~77 lines |
| **Transition glitch** | `0x60` (single occurrence) | 1164 | 1 line |

**All unique non-idle byte values from teraterm.txt:** `0x33`, `0x38`, `0x60`, `0x67`, `0x77`

**Note:** TeraTerm mangles bytes ≥0x80 via UTF-8 (`FE`→`EF BF BD`) and strips
control chars (`06`, `00`). Only `0x30`, `0x70`, and bytes <0x80 survive intact.
The non-idle bytes above are all <0x80 so they're reliable.

### Pin 6 (Green) — Display Refresh Stream

Continuous repeating pattern, no variation on button presses:
```
77 E6 E6 77 E6 E6 77 E6 E6 FF
```
Likely the multiplexing/refresh signal for the 7-segment display. Captured via
RS-485 adapter on Green pair (A=Green/White, B=Green, GND=Blue).

**Caution:** Connecting the RS-485 adapter to this pair caused display corruption
(panel showed "80" flashing). The adapter loads the line and interferes with
display refresh.

## What We Don't Know Yet

1. **7-segment lookup table** — Brute-force found 72 candidate solutions (2-digit, 81-95°F),
   but user confirms tub was set to 104°F during capture. Balboa panels show **actual
   water temperature** in idle mode (not setpoint), so idle display may have been lower
   than 104 if the tub was still heating. Alternatively, the display might be 3-digit.
   **Need temperature ladder capture at known temperatures to resolve.**
2. **Display digit count** — Is the VL panel 2-digit or 3-digit? How does it display
   temperatures ≥100°F? (small "1" indicator LED? alternating? 3 full digits?)
3. **Full display protocol structure** — 8-byte idle frame, 11-byte burst frame, and
   7-byte lights frame all need structural understanding. Frame boundaries and byte
   roles (digit vs LED indicator vs control) are partially mapped but not confirmed.
4. **What the display modes mean** — Three non-idle modes observed in teraterm.txt:
   button-press (0x77), possible Pr/priming (0x67 0x33), and transition (0x60).
   Need captures during known display states to confirm.
5. **ESP32 circuit design** — analog switch IC selection, voltage levels,
   impedance matching for button injection
6. **Whether we need to read the RS-485 streams** — if we only need to inject
   button presses (temp up/down), we may not need to decode the display protocol
   at all. But reading current temp from the display stream would enable
   closed-loop control.

## 7-Segment Analysis Status

### Reliable Pin 5 Data Points

| Source | Byte 3 | Full idle frame | Notes |
|--------|--------|-----------------|-------|
| `hex.txt` (TeraTerm) | `0x30` | `FE 06 70 30 00 06 70 00` | ~1350 stable frames, 5 sporadic `0x38` (noise) |
| `rs485_jets.txt` (Python) | `0xE6` | `FE 06 70 E6 00 06 70 00` | 285 frames, rock-solid, different session than hex.txt |

Both captures were on Pin 5 via RS-485 adapter. The temperature changed between sessions.
Only byte index 3 differs.

**Important context:** User confirms tub setpoint was 104°F during capture sessions.
Balboa panels display actual water temperature in idle mode (not setpoint). If the tub
was heating, the idle display could have shown a lower actual water temp (e.g., 81-82°F).
Setpoint is only shown briefly after pressing temp up/down.

### Additional Pin 5 Observations

- **Button-press burst** (hex.txt frame ~1346): `E6 77 30 E6 77 E6 E6 77 E6 E0 FF`
  - 11 bytes, embeds the current digit byte (`0x30`) at position 2
  - Other unique bytes in burst: `0x77`, `0xE0`, `0xFF`
- **Sporadic `0x38`** at byte 3 in hex.txt (5 occurrences among ~1350 frames) — likely
  noise or bit errors during capture, not a real temperature value
- **Pin 6 refresh** (rs485_capture.txt): `77 E6 E6 77 E6 E6 77 E6 E6 FF` — no variation
- **OH.csv logic analyzer**: 3 bytes per burst (`E6 E6 06`) on CH4 — unreliable byte
  values due to single-ended capture of differential RS-485, but timing is solid
  (500ms on / 500ms off flash, 60Hz refresh during on-phase)

### Key Constraints
- Only byte index 3 changes between the two Pin 5 captures: `0x30` ↔ `0xE6`
- `0x30` has 2 bits set → must be digit "1" (the only 2-segment digit)
- `0xE6` has 5 bits set (all 8 bits) → could be "2", "3", or "5" if bit 7 is a segment
- Mathematical proof: `0x30`="1" and `0xE6`="4" CANNOT coexist under any 7-bit mapping
  (segments b,c shared by "1" and "4" require 2 shared bits, but `0x30` and `0x66` share only 1)

### Brute-Force Results
- **v1** (bits 0-6 only, bit 7=dp, 3-digit temps 80-106°F): **0 solutions**
- **v2** (any 7 of 8 bits, dp can be any bit, 2- or 3-digit temps 50-120°F):
  **72 solutions** across three temperature pairs:
  - 81↔82°F — `FE`="8", `0x30`="1", `0xE6`="2" (most likely)
  - 81↔85°F
  - 91↔95°F
- No 3-digit temperature solution exists with only 2 data points — the search space
  is too unconstrained

### Exhaustive Mapping Search (analyze_protocol.py)
Tested all P(8,7)=40,320 mappings × 8 dp positions against all known byte values.
Scored by decode coverage, temperature validity, and consistency.

**Top result (score 200):** `a:6, b:5, c:4, d:7, e:2, f:3, g:1`, dp=bit0
- `0xFE`="8", `0x30`="1", `0xE6`="2", `0x70`="7", `0x06`="r", `0xF3`="3", `0x03`="-"
- Frame A decodes: `[8 r 7 1] [_ r 7 _]` → 81°F (if 2-digit, bytes 0+3)
- Frame B decodes: `[8 r 7 2] [_ r 7 _]` → 82°F
- `0x06`="r" at bytes 1,5 likely represents LED indicators, not a digit
- `0x77` (button-press mode) does NOT decode to any standard character under this mapping
- Temp-down frame byte 7: `0xF3`="3" — possibly new setpoint digit

**Caveat:** This result assumes 2-digit temperature display. If the actual water temp
was 104°F (not 81°F), the mapping is wrong and we need more data points.

### Reference Projects
- **MagnusPer/Balboa-GS510SZ** (GS510SZ + VL801D): bit6=a, bit5=b, bit4=c, bit3=d, bit2=e, bit1=f, bit0=g
  - Confirmed: "1"=0x30, "7"=0x70, "0"=0x7E — matches our `0x30` and `0x70`
  - Does NOT match `0xE6` — our VS300FL4 may use bit 7 as a segment bit (not dp)
- **netmindz/balboa_GL_ML_spa_control**: Uses ASCII encoding for ML-series "smart" panels —
  different protocol family, NOT applicable to our VL "dumb" panel

### Phase 1.5: RS-485 Capture Session (NEXT)
**Goal:** Collect enough Pin 5 data at known temperatures to solve the 7-segment encoding.

**Equipment:** RS-485 adapter (MAX485/ESP32) on Pin 5, Python capture script

**Capture checklist:**

1. **Determine display digit count** — Look at the panel while it's showing temperature.
   Is it 2 digits? 3 digits? Does it have a separate "1" LED for hundreds?

2. **Note the actual water temperature** — Before capturing, let the tub reach a
   stable temperature. Note what the display shows. This is the ACTUAL water temp
   (not setpoint).

3. **Temperature ladder capture** — Capture idle frames at 5+ known temperatures.
   Use `scripts/rs485_capture.py` or a simplified version. For each temperature:
   - Wait for display to show stable idle temperature
   - Capture ~10 seconds (enough for 50+ frames)
   - Save to `pin5_XXX.txt` where XXX is the displayed temperature
   - Press temp down once, wait for display to settle, capture again
   - Repeat: 104→103→102→101→100→99 (or whatever range is convenient)

4. **Setpoint display capture** — Press temp up or temp down and capture during
   the brief setpoint flash (display mode with `0x77` bytes). Note what temp
   the setpoint is showing.

5. **Pr mode capture** (if possible) — If you can trigger the priming display,
   capture during it. This should show the `0x67 0x33` pattern.

**With 5+ temperature data points, the brute-force script can uniquely solve the
segment-to-bit mapping and we can build a complete lookup table.**
