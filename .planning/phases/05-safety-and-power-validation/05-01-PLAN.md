## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Sensor -> Automation | Temperature and setpoint values from ESP32 could be stale, unknown, or unavailable |
| Automation -> Automation | Thermal runaway sets flag that TOU reads; race conditions possible |
| Auto-clear -> Flag | Premature flag clear could allow TOU to raise setpoint during active overshoot |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | ha/thermal_runaway.yaml | mitigate | Safe float defaults: float(0) for temp (won't trigger overshoot), float(999) for setpoint (won't trigger overshoot). Condition block gates on unknown/unavailable. |
| T-05-02 | Denial of Service | ha/tou_automation.yaml + ha/thermal_runaway.yaml | mitigate | input_boolean.thermal_runaway_active coordination flag: moderate/severe set it, TOU checks it before raising setpoint. Severe also disables TOU entirely. |
| T-05-03 | Tampering | ha/thermal_runaway_clear.yaml | mitigate | Auto-clear requires 2-minute sustained temp <= setpoint AND flag must be "on". Does NOT re-enable TOU automation. Severe events require manual re-enable. |
| T-05-04 | Information Disclosure | ha/thermal_runaway.yaml | mitigate | Gate on binary_sensor.tublemetry_hot_tub_api_status == "on" prevents acting on stale data from offline ESP32. |
</threat_model>
