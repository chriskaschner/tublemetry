Build the ESPHome external component that reads the VS300FL4's RS-485 display stream and surfaces the current water temperature as a read-only climate entity in Home Assistant, with diagnostic sensors.

**Python config files (ESPHome component registration):**

1. `__init__.py` -- Main component config schema:
   - Define CONF keys for uart_pin5, uart_pin6
   - CONFIG_SCHEMA: cv.Schema with two required UART parent IDs

2. `climate.py` -- Climate platform:
   - CONFIG_SCHEMA with required `tubtron_id` (references parent component)
   - `async def to_code(config)`: Get parent, create climate, register

3. `sensor.py` -- Numeric sensor platform:
   - CONFIG_SCHEMA with `decode_confidence` sensor config (entity_category: diagnostic)
   - `async def to_code(config)`: Create sensor, set on parent

4. `text_sensor.py` -- Text sensor platform:
   - CONFIG_SCHEMA with `display_string`, `raw_hex`, `display_state`, `digit_values`, `last_update` text sensor configs (all entity_category: diagnostic)
   - Per user decision: `digit_values` exposes per-digit breakdown (e.g. "1|0|4" or JSON "[\"1\",\"0\",\"4\"]") as a separate diagnostic text sensor. This implements the locked requirement "per-digit breakdown as attributes" -- using a dedicated text sensor rather than entity attributes because ESPHome text_sensor attributes are limited. The per-digit text sensor provides the same data in a parseable format.
   - Per user decision: `last_update` exposes a timestamp of the last display frame processed. This implements the locked requirement "last update timestamp" as a diagnostic text sensor publishing ISO 8601 formatted time from the ESPHome `sntp` time component.
   - `async def to_code(config)`: Create text sensors, set on parent via setters (set_digit_values_sensor, set_last_update_sensor, etc.)

**C++ files (core logic):**

5. `tubtron_display.h` -- Class declaration:
   - TubtronDisplay inherits from: climate::Climate, Component
   - Store two UARTDevice pointers (pin5_uart_, pin6_uart_) -- NOTE: do NOT inherit from UARTDevice since we have two UARTs. Instead, store uart::UARTComponent pointers and call available()/read_byte() on them directly.
   - Private members: pin5_buffer_[16], pin5_index_, pin5_last_byte_time_, last_temperature_ (NAN), last_display_string_, last_display_state_, last_raw_hex_, last_digit_values_
   - Sensor/TextSensor pointers for diagnostic outputs (set via setters): display_string_sensor_, raw_hex_sensor_, display_state_sensor_, decode_confidence_sensor_, digit_values_sensor_, last_update_sensor_
   - Public: setup(), loop(), traits(), control(), get_setup_priority()
   - Public setters: set_digit_values_sensor(text_sensor::TextSensor*), set_last_update_sensor(text_sensor::TextSensor*)
   - Private: process_pin5_data(), decode_pin5_frame(), decode_7seg(), update_temperature(), is_valid_temperature()

6. `tubtron_display.cpp` -- Implementation:
   - Port SEVEN_SEG_TABLE from Python's decode.py to a static C array or switch statement. Include confirmed mappings (0x30->"1", 0x70->"7", 0x00->" ") and GS510SZ reference mappings marked with comments as unverified.
   - IMPORTANT: Format the C++ table in a parseable way so the cross-check test can extract mappings. Use a consistent format like: `{0x30, '1'}, // confirmed` -- one entry per line, hex value and character clearly visible.
   - `decode_7seg(uint8_t)`: Mask bit 7, lookup in table, return char ('?' for unknown).
   - `setup()`: Log startup, set update interval to 0 (loop-driven).
   - `loop()`: Call process_pin5_data(). Optionally process Pin 6 data if the second UART is configured.
   - `process_pin5_data()`: Read bytes from pin5_uart_. Use timing-based frame boundary detection (>1ms gap = new frame). Buffer into pin5_buffer_. When 8 bytes collected, call decode_pin5_frame().
   - `decode_pin5_frame(const uint8_t*, size_t)`: Decode each byte via decode_7seg. Build display_string. Check for FE marker. Build raw_hex string. Build digit_values string (pipe-separated per-digit characters, e.g. "1|0|4"). Call update_temperature(). Update diagnostic sensors only on change. When any value changes, also publish to last_update_sensor_ with current timestamp (use `id(sntp_time).now().strftime("%Y-%m-%dT%H:%M:%S")` or `millis()` as fallback if SNTP not configured).
   - `update_temperature(const string&)`: If valid 2-3 digit number in 80-120 range, update current_temperature and publish_state(). If non-temperature (OH, ICE, --), keep current_temperature unchanged, update display_state sensor. This is the dumb decoder pattern -- no interpretation.
   - `traits()`: supports_current_temperature(true), supported_modes({HEAT}), visual_min(80), visual_max(104), visual_step(1). set_supports_action(false).
   - `control(const ClimateCall&)`: Phase 1 is read-only. Log that control is not supported yet. Do nothing.
   - `get_setup_priority()`: Return setup_priority::DATA (after UART is ready).

   Key implementation notes:
   - Use `millis()` not `micros()` for frame timing (>1ms gap detection). At 115200 baud, one byte = 87us, so 1ms gap = ~11 byte-times, plenty of margin.
   - For raw_hex: format as "FE 06 70 30 00 06 70 00" (space-separated uppercase hex).
   - For digit_values: format as pipe-separated decoded characters, e.g. "F|_|7|1| |_|7| " where each position maps to its decoded character. This gives HA full per-digit visibility.
   - Only publish to HA on change: compare new values against last_* members before calling publish_state() or sensor->publish_state().
   - Include ESP_LOGD/ESP_LOGW for debug and warning logging throughout.

**Cross-check test (verification bridge):**
