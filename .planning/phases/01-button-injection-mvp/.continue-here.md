<decisions_made>

- Protocol is synchronous clock+data (like SPI), NOT RS-485 UART
- Pin 6 (Green, T568B) = Clock signal, 24 pulses per frame at 60Hz
- Pin 5 (Blue/White, T568B) = Data signal, sampled on clock rising edge
- Pin 4 (Blue, T568B) = Ground reference (confirmed NOT board chassis ground, but circuit ground for 5V rail)
- Frame structure: 3 x 7-bit display digits + 3 status bits = 24 bits
- GS510SZ 7-segment encoding confirmed correct: 0x7E="0"/"O", 0x37="H", 0x00=blank
- RS-485 adapter is WRONG TOOL -- multiple sessions wasted, all wiring combos tried and documented
- MagnusPer/Balboa-GS510SZ is the correct reference project (not netmindz/balboa_GL_ML_spa_control which is for ML smart panels)
- ESP32 firmware approach: attachInterrupt on clock pin, digitalRead data pin on each edge (MagnusPer approach)
- OR-gate buffer recommended for protection when tapping signal
- Logic analyzer (Lonely Binary 8ch + sigrok-cli) is the right capture tool
</decisions_made>

<context>
The big insight this session: we spent time trying every RS-485 adapter wiring combination
on macOS (all documented in memory). Nothing worked correctly because the protocol isn't
RS-485 UART at all. Web research on MagnusPer/Balboa-GS510SZ revealed the VL "dumb" panel
uses synchronous clock+data protocol. Re-analyzing existing logic analyzer captures (OH.csv)
with this understanding produced perfect decodes.

The VS300FL4 protocol is simpler than GS510SZ: 24 bits per frame (vs 39), 3 digits (vs 4+),
3 status bits. The 7-segment encoding is identical.

For the logic analyzer capture session at the tub:
1. Connect: CH0=Pin 5 (Blue/White, data), CH1=Pin 6 (Green, clock), GND=Pin 4 (Blue)
2. Use stub cable with all pins through T-splitter at J1
3. sigrok-cli: sigrok-cli -d fx2lafw -c samplerate=1000000 --samples 10000000 --channels 0,1 -o capture.sr
4. Capture steady temp first, then press temp down and capture at each step
5. Export: sigrok-cli -i capture.sr -O csv > capture.csv
6. Decode: uv run python 485/scripts/decode_clockdata_v2.py capture.csv

WARNING: previous sessions noted T-splitter + analyzer loading Pin 6 can cause display
corruption. If display goes weird, disconnect Pin 6 probe and capture Pin 5 only.
</context>
