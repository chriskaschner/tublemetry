## Naming Conventions

**Files:**
- Capture data: `rs485_capture.txt`, `rs485_jets.txt`, `teraterm.txt` (descriptive, lowercase with underscore)
- Logic analyzer: `254.csv`, `OH.csv`, `254.sr` (hardware or channel identifiers)
- Baud rate tests: `{BAUD}.txt`, `{BAUD}.log` (baud value as filename)
- Scripts: `{action}_{target}.py` (e.g., `rs485_capture.py`, `bruteforce_7seg.py`, `decode_csv.py`)

**Directories:**
- Main work: `485/` (board model identifier)
- Subdirectories: lowercase plural for collections (`scripts/`, `reference/`)
- Planning: `.planning/` (hidden, prefixed with dot)

**Python Scripts:**
- File naming: lowercase_with_underscores.py
- Module docstrings: Describe purpose, input files, and approach (see first lines of each script)
- Constants: UPPERCASE (PORT, BAUD, SAMPLE_RATE, BAUD_RATES, CSV_PATH)
- Functions: snake_case with descriptive names (decode_byte, find_bursts, test_mapping, uart_decode_region)

**Data Files:**
- Hex dumps: Space-separated uppercase hex (e.g., `FE 06 70 E6 00 06 70 00`)
- Timestamps: Floating-point seconds since capture start, 3 decimal places (e.g., `[123.456]`)
- Markers: `===== MARKER N: [button_name] at TIME =====` format

## Where to Add New Code

**New Capture Tool:**
- Primary code: `485/scripts/rs485_{description}.py`
- Pattern: Import `serial`, set PORT and BAUD constants, open serial port, read bytes in loop, save to `{description}.txt`
- Template: Copy structure from `rs485_capture.py` or `rs485_scan.py`
- Output location: Save to `485/{description}.txt` (co-located with other captures)

**New Analysis Script:**
- Primary code: `485/scripts/{action}_{source}.py`
- Pattern: Read from `485/{input_file}.txt` or `.csv`, perform analysis, print results to stdout and optionally file
- Data input: Load bytes from hardcoded `485/` paths; use standard frame definitions (FRAME_A, FRAME_B, FRAME_TEMP_DOWN, etc.)
- Example patterns:
  - To analyze captures: `decode_csv.py` (read CSV, UART decode), `analyze_protocol.py` (pattern matching)
  - To search mappings: `bruteforce_7seg.py` (exhaustive search with scoring)

**New Decoder/Mapping:**
- Primary code: `485/scripts/bruteforce_{target}.py`
- Pattern: Define character lookup (DIGIT_SEGMENTS, LETTER_SEGMENTS), implement scoring function, exhaustive search with ranking
- Known reference frames: Hardcode Frame A/B/Temp variations at script top (see `bruteforce_7seg.py` lines 63-68)
- Output: Print best mappings to stdout, optionally save detailed decode results

**Hardware Documentation:**
- Primary: `485/README.md` (protocol summary, confirmed parameters, observations)
- Status: `485/rs485-status-{date}.md` (investigation checkpoints)
- Research: `WORKLOG.md` (decisions, research findings, community context)

## File Organization Rationale

**Flat script organization (`485/scripts/`):**
- Rationale: Each script is independent analysis tool; no shared Python modules needed yet
- Benefit: Easy to run individual scripts, no import path management
- Trade-off: Some code duplication (e.g., SEGMENT definitions, decode functions)
- Future: If tools grow, consider `lib/` subdirectory for shared utilities (decode logic, frame definitions)

**Capture data in root (`485/`):**
- Rationale: Single location for all data artifacts makes backup/sharing easier
- Benefit: Can easily see file sizes, timestamps, and manage large exports
- Trade-off: Can become cluttered; recommend archiving old captures
- Naming: Descriptive but not nested; timestamp in filename if multiple versions

**Reference research external (`reference/`):**
- Rationale: Keeps codebase focused on Balboa VS series; external docs separate
- Benefit: Clear boundary between original research and community reference
- Trade-off: May need to cross-reference; consider symbolic links or search tool
